# 竖版废土坦克大战首关设计稿

## 1. 目标与范围

本设计稿用于定义 `tankwar` 项目的首关资源设定、资源组织方式、首关节奏和最小可玩闭环，目标是在 WebGL 移动浏览器场景下，先做出一关可完整通关的 2D 竖版废土坦克战。

本轮只覆盖首关设计，不覆盖以下内容：

- 第二关及后续关卡扩展
- 长线成长、货币、商店、数值养成
- 联机、排行榜、任务系统
- 复杂随机刷怪系统
- 非首关必要的高级武器分支

## 2. 已确认决策

### 2.1 核心定位

- 游戏类型：WebGL 移动浏览器为主的 2D 竖版坦克大战网页游戏
- 主题方向：废土题材
- 玩法参考：类似《赤色要塞》的纵向推进式街机体验
- 当前目标：先完成一关，从开场推进到关底 Boss 击破的完整流程

### 2.2 视觉与镜头

- 首关主骨架：`中轴公路突进`
- 镜头视角：`轻 3/4 俯视`
- 美术颗粒度：`清晰街机废土`
- 美术要求：
  - 地图与元素要有明确街机感
  - 读图优先，轮廓优先，小屏识别优先
  - 保留铁锈、沙尘、废弃感，但不做脏乱写实堆料

### 2.3 主角与操作

- 主角定位：`中型坦克`
- 移动速度：`中速`
- 射击节奏：`中速`
- 防御表现：使用 `血条`
- HUD 约束：`主角和敌人都要有血条`
- 操作方案：
  - 左侧 `半透明摇杆` 控制移动
  - 右侧 `半透明开火按钮` 控制主射击

### 2.4 敌人与推进方式

- 首关敌人编制：
  - 轻坦
  - 中坦
  - 固定炮台
  - 侦察直升机
  - 关底 `移动堡垒型 Boss`
- 推进方式：`混合式`
  - 普通路段：玩家推进式上滚
  - Boss 段：进入固定锁场战斗

### 2.5 掉落

- 首关掉落方案：`B`
- 掉落种类限定为：
  - `repair_kit` 修理包
  - `weapon_boost` 临时双发/强化炮
  - `shield_battery` 护盾电池

## 3. 地图与场景资源 Contract

### 3.1 地图管线选择

首关地图采用以下资源轴：

- `visual_model = layered_raster`
- `runtime_object_model = separate_props`
- `collision_model = coarse_shapes + trigger_zones`
- `engine_target = project-native WebGL 2D`
- `art_style = clean_hd + arcade readability + wasteland theme`

### 3.2 选择理由

这套方案的目标不是做编辑器优先的 tilemap，而是优先保住“清晰街机废土公路”的观感，同时保留首关和后续关卡对障碍、掩体、刷怪点、Boss 场地的调整自由度。

具体约束如下：

- 底图只负责地表关系，不把废车、油桶、沙包等障碍直接焊死在地面图里
- 路障、废车、门架、炮台基座等场景元素全部独立为透明 prop
- 碰撞优先使用矩形/圆形等粗碰撞，不做像素级碰撞
- 刷怪、掉落、锁场、Boss 触发通过 zone 管理，不靠硬编码散落在运行时逻辑中

### 3.3 地图分段

首关地图不做一张超长大图，而是拆成四段竖版切片：

1. `segment-01` 开场推进段
2. `segment-02` 路障与炮台压制段
3. `segment-03` 检查站封锁段
4. `segment-04` Boss 固定战场段

### 3.4 场景道具范围

首关独立场景 prop 需至少包含：

- 废车
- 油桶
- 沙包
- 路障
- 铁丝网
- 检查站门架
- 炮台基座
- 重卡残骸
- 混凝土掩体

## 4. 战斗资源 Contract

### 4.1 资源拆分原则

首关战斗资源不以复杂逐帧人物动画为主，而是以“车体位移 + 炮塔朝向 + 开火反馈 + 命中特效 + 残骸态”构成主要表现力。

因此，主坦克、敌方坦克、Boss 均优先采用分层或分部件结构，而不是单张大图烘死全部表现。

### 4.2 主角中型坦克包

主角资源至少包含：

- `hull`
- `turret`
- `muzzle_flash`
- `hit_flash`
- `wreck`
- `shadow`
- `dust_fx`

实现约束：

- 车体与炮塔必须分层
- 炮口表现要能支持主开火反馈
- 残骸态要明显区别于正常态

### 4.3 普通敌军包

#### 轻坦

- `base`
- `fire`
- `hit`
- `wreck`
- `shadow`

要求：轮廓更小、更灵活，阅读上明显区别于中坦。

#### 中坦

- `base`
- `fire`
- `hit`
- `wreck`
- `shadow`

要求：车体更厚、炮更粗、血量表现更重。

#### 固定炮台

- `base`
- `turret`
- `fire`
- `hit`
- `wreck`
- `shadow`

要求：底座和炮头拆开，便于炮头偏转。

#### 侦察直升机

- `body`
- `rotor`
- `fire`
- `hit`
- `crash`
- `shadow`

要求：空中单位轮廓必须与地面单位完全区分，不能在小屏上混为一类。

### 4.4 Boss 资源包

Boss 为 `移动堡垒型`，采用分部件结构：

- `body`
- `main_cannon`
- `side_cannon_left`
- `side_cannon_right`
- `weakpoint_core`
- `damage_overlay`
- `wreck_large`

设计约束：

- Boss 不能只是普通坦克放大版
- 必须有“分部件可读性”
- 弱点和受损阶段要有明确视觉差异

### 4.5 掉落与战斗特效

#### 三种掉落

- `repair_kit`
- `weapon_boost`
- `shield_battery`

每种掉落至少包含：

- `idle`
- `glow`
- `pickup_fx`

#### 公共战斗特效

- 小爆炸
- 中爆炸
- Boss 大爆炸
- 命中火花
- 炮弹拖尾
- 护盾受击闪烁

### 4.6 HUD 与战斗 UI

首批 HUD 需包含：

- 主角血条
- 普通敌人头顶血条
- Boss 大血条
- 护盾条或护盾覆盖表现
- `weapon_boost` 状态图标
- 左摇杆底座
- 左摇杆摇杆头
- 右侧半透明开火按钮

风格约束：

- 半透明
- 军武街机感
- 干净、轻、透
- 不做厚重写实仪表风
- 不做花哨二次元 UI

## 5. 首关节奏与关卡 Contract

### 5.1 总体结构

首关结构为：

- `4` 段推进
- `1` 个固定 Boss 场

整体目标体验：

- 普通推进约 `3-4` 分钟
- Boss 战约 `1-2` 分钟
- 首通总时长约 `5-6` 分钟

### 5.2 Segment-01 开场推进段

目标：

- 玩家在 `5-8` 秒内进入第一次稳定交火
- 不做教学区，直接进入核心节奏

建议内容：

- 中轴公路主路面
- 左右少量废车、油桶
- 第一波以轻坦为主
- 一处偏侧翼固定炮台

### 5.3 Segment-02 路障与压制段

目标：

- 抬高强度
- 引入中坦与空中骚扰

建议内容：

- 更明显的废车、沙包、断裂路障
- 中坦首次登场
- 左右配合的固定炮台
- 侦察直升机首次入场

### 5.4 Segment-03 检查站封锁段

目标：

- 形成短促而明确的半锁场压迫
- 在 Boss 前做补给过渡

建议内容：

- 检查站门架、铁丝网、障碍车构成封锁口
- 轻坦、中坦、炮台混合编制
- 直升机从侧上方切入一次形成交叉火力
- 结束后稳定给一次正向掉落

掉落约束：

- 优先掉 `repair_kit` 或 `weapon_boost`
- 首关先偏脚本化，不做高随机

### 5.5 Segment-04 Boss 固定战场

目标：

- 与普通推进段形成明确节奏落差
- 进入固定锁场 Boss 战

场地要求：

- 更宽的废土公路终点战场
- 左右有战场边界元素，如重卡残骸、混凝土掩体、炮台残骸
- 中间保留足够机动空间
- 入口关闭，镜头锁场

### 5.6 Boss 节奏

Boss 采用两阶段弱点节奏：

#### 第一阶段

- Boss 主体推进
- 左右副炮提供压制

#### 第二阶段

- 当副炮被打掉，或 Boss 掉血到阈值后
- 暴露核心弱点
- 火力模式升级

设计目标：

- 避免纯堆血条
- 强化街机 Boss 识别感
- 让“移动堡垒”区别于普通重坦

### 5.7 刷怪与掉落规则

首关优先采用强控制脚本刷怪：

- 普通推进段按 `trigger zone` 刷怪
- 关键敌人清掉后，下一波再入场
- 掉落先走半固定，不做重随机

这样做的原因是：

- 首关节奏可控
- 易于调试
- 不会因为随机性把体验打散

## 6. 推荐资源管线与落地方案

三种候选方案中，最终选定：

- `方案 B：分层底图 + 独立场景道具包`

不采用其他两种方案的原因：

- 不选整张大底图方案：后续调图、调路障、调 Boss 区返工成本高
- 不选 tilemap 方案：首关成本高，且容易削弱废土街机质感

方案 B 的价值：

- 首关更容易先做出观感
- 地图与场景调整自由度高
- 第二关以后可以直接复用同一套路面、prop、zone 结构

## 7. 首批生成顺序

为尽快形成首关完整闭环，资源生成顺序固定为：

1. `segment-01 ~ 04` 地图底图
2. 场景 prop 包
3. 主角中型坦克
4. 轻坦
5. 中坦
6. 固定炮台
7. 侦察直升机
8. 三种掉落
9. HUD
10. Boss 移动堡垒
11. 公共爆炸/命中特效
12. 全关拼接预览与资源清单校验

顺序原则：

- 先保证普通推进段可玩
- 再补 Boss 和 HUD
- 最后补全公共特效与总预览

## 8. 非目标与后续边界

以下内容明确不纳入首关第一轮交付：

- 多武器树和复杂升级体系
- 随机地图生成
- 大量步兵或多兵种扩展
- 无尽模式
- 高复杂度掉落池
- 多关卡共享编辑器系统

这些内容可以在首关跑通后，基于现有 contract 再逐步扩展。

## 9. 最终结论

首关的正确落地方向已经明确：

- 玩法上采用 `混合式推进`
- 地图上采用 `layered_raster + separate_props`
- 单位上优先走 `分层坦克 / 分部件 Boss`
- 美术上采用 `清晰街机废土`
- 目标是先做出 `5-6` 分钟的一关闭环，而不是提前建设重型系统

这份设计稿可直接作为后续资源生成、关卡 JSON 设计、HUD 落地和首关实现计划的基线文档。
